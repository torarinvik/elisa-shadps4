# Agent E3 — Ord Protocol For Address-Ordered Tables

**Status:** COMPLETE (lawful base-key `Ord` added; sorted invariants covered by focused runtime verification)

## Fit Finding

- Real ordered surfaces found:
  - `MemoryManager_InsertDmemArea` maintains `dmem_areas` in base-address order for direct-memory allocation/query.
  - `Linker_NextModuleBase` and `Linker_FindModuleContainingAddress` consume module base/extent ordering expectations.
- Whole-record `Ord` for `PhysicalMemoryArea` / `Module` was intentionally avoided: base-only ordering is not antisymmetric under full-record equality when non-key fields differ.
- Added explicit base-key types instead:
  - `PhysicalMemoryAreaBaseKey`
  - `ModuleBaseKey`

## Verification Notes

- `protocol Ord` with total/antisym/transitive laws parses, but this `elisac` build does not statically refute a deliberately broken comparator in a throwaway probe (`le` returning `true` for all `u64` values compiled and ran). Broken-comparator coverage is therefore by concrete law/property-style runtime tests.
- Struct `==` on the key structs triggered an LLVM ICmp type error, so the runtime antisymmetry checks compare the single `base` field directly.
- A postcondition form such as `ensure PhysicalMemoryArea_BaseKeysSorted(self.dmem_areas)` was attempted and rejected because region-polymorphic sorted checks cannot be inferred/discharged at that function exit today. The invariant is checked in the focused test target.

## Target Stanza

  - Adds `kernel_memory_ord_base_key_laws_and_sorted_invariants`
  - Covers `Ord` law checks for region/module base keys and sorted invariant checks for `dmem_areas` and module tables.

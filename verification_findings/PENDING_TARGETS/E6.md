# E6 — Audio / Pad / Network HLE Boundary Contracts + Gate Integration

## Contracts added

### core/libraries/audio/audioout.elisa

No `requires`/`ensure` added in the final commit.

**Rationale:** `sceAudioOutOpen` error-path tests (line 78–79 in `elisa_tests/core_libraries_audio_parity_tests.elisa`) intentionally call with `length=0` and `sample_rate=44100` to verify error returns.  Adding `requires length > 0 and length <= 2048` / `requires sample_rate == 48000` would make those call sites statically unprovable, breaking the test.  The internal body guards (early-return `if length == 0`, `if sample_rate != 48000`) already enforce these at runtime.  Arithmetic `requires` are only sound when the parity-test suite does NOT exercise out-of-range inputs; that condition does not hold here.

### core/libraries/audio/audioin.elisa

No `requires`/`ensure` added in the final commit (same rationale: error-path test coverage exercises out-of-range inputs via `sceAudioInOpen`).

### core/libraries/pad/pad.elisa

| Function | Guard added |
|---|---|
| `scePadRead` | `requires num > 0` |

- `scePadReadState`, `scePadGetControllerInformation`, `scePadGetInfo`, `scePadSetLightBar`, `scePadSetVibration`: pointer null-checks were considered but removed because the parity suite explicitly calls `scePadSetLightBar(h, null)` to test the error path (elisa_tests/core_libraries_pad_tests.elisa:32).  The body guards already return `ORBIS_PAD_ERROR_INVALID_ARG` for null.

### core/libraries/network/net.elisa

| Function | Guard added |
|---|---|
| `sceNetBind` | `requires addrlen > 0` |
| `sceNetRecv` | `requires len > 0` |
| `sceNetSend` | `requires len > 0` |
| `sceNetSendto` | `requires len > 0` |

These functions are NOT called directly from the parity test (net tests go through the C FFI bridge), so the prover does not generate unprovable static warnings at test call sites.  The contracts reflect the POSIX requirement that `addrlen` and buffer `len` are positive when calling `bind`/`send`/`recv`.

### core/libraries/network/netctl.elisa

No `requires`/`ensure` added in the final commit.

**Rationale:** All netctl functions whose bodies check for null (`sceNetCtlGetInfo`, `sceNetCtlGetState`, `sceNetCtlGetResult`, `sceNetCtlRegisterCallback`) are called from the parity test with stack-allocated structs.  The prover cannot discharge `stack_local != null` statically, so pointer-null `requires` contracts would generate "could not be proven statically" warnings at every test call site.

## True unresolved obligations

No new `hle-NN.md` obligations were generated by E6.  The pre-existing proof gaps in `MemoryManager_MapMemory`/`Module_AlignUp` are tracked in existing findings.

The following is an E6-discovered observation (not a new file): the `sceNetCtlRegisterCallback` and `sceNetCtlRegisterCallbackForNpToolkit` functions check `func == 0` but accept a raw `i64` cast to a function pointer type.  A `requires func != 0` contract is the correct boundary, but the prover cannot discharge it from opaque `i64` values at call sites.  This is noted for a future pass that adds wrapper types or refinement narrowing on the callback `func` parameter.
